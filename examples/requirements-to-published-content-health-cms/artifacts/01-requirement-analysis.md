## 1. 业务目标与干系人

- **目标**：从分散在网盘、旧后台、公众号的内容，收敛到**统一内容中台**，支撑医学科普、患教、产品说明等场景的**生产、审核、发布与度量**。  
- **关键干系人**：医学部（审核主体）、市场/运营（生产与渠道）、IT（集成与账号）。  
- **非本期范围**：直播业务仍托管在第三方，媒资不强制迁入。

---

## 2. 需求优先级（推断）

| 维度 | 判断 |
|------|------|
| **P0** | 权限与内容生产、可审计发布路径、版本与定时、基础搜索、DAM、日志、对接 API 的**可行性** |
| **P1** | 数据看板、SSO、标签体系统一与长期运营 |
| **可分期** | 全链路「自定义 BPM」、全渠道一键分发、深度自定义元数据模型、收藏/批量下载等——需按产品与集成评估 |

---

## 3. 约束与澄清点

1. **审核**：仅需医学部参与，流程不宜复杂成多部门串行；与「可配置多节点 SOP」需区分——客户要**效率**还是**可证明的合规链**。  
2. **规模**：5000+ 条、40G+，需关注**导入、存储、检索性能**与**归档策略**。  
3. **搜索**：首期标题 + 标签；「疾病领域分布」等看板依赖**分类/标签建模**是否一致。  
4. **集成**：集团内系统对接依赖 **API 能力与安全策略**；账号不接 OA 降低首期集成面。  
5. **版权**：到期自动下架与归档——需明确「版权字段」与「到期」数据来源（手工维护 vs 系统计算）。

---

## 4. 风险摘要

| 风险 | 说明 |
|------|------|
| 期望为「全能 ECM + BPM」 | 需对齐产品边界，避免范围蔓延 |
| 看板与「领域分布」 | 依赖内容打标质量，否则指标失真 |
| 多渠道一键发布 | 往往需**分渠道连接器**与分阶段交付 |
