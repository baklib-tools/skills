# 正式方案书（示例）：基于 Baklib 构建康维医疗内容中台

**文档性质**：售前技术方案（示例），用于说明**响应思路与分期**；**不构成合同承诺**，实施以签约与产品版本为准。  
**依据**：客户需求见 [fixtures](../fixtures/requirement-input-desensitized.md)；产品能力须以贵司 **`baklib-workspace`** 内最新说明与官方文档为准。

---

## 一、综述

建议采用 **Baklib** 作为内容与站点/知识发布核心平台，覆盖**知识库、资源库（DAM）、站点化发布、Open API 集成、操作审计**等能力；对「高度可编排审批链」「全渠道一键矩阵」等诉求，采用 **分阶段：先标准能力落地，再集成扩展**。

---

## 二、需求响应总表（摘要）

| 类别 | 总体响应 |
|------|----------|
| 文章与权限、版本、定时、分享链接 | **可响应**（产品核心能力范围内配置） |
| 标签、搜索（标题+标签）、导入与批量 | **可响应 / 部分响应**（高级检索与复杂目录策略可能分期） |
| DAM、操作日志、成员与角色 | **可响应** |
| 标准 API 与对内集成 | **可响应**（按 Open API 与网关策略实施；字段以实际版本为准） |
| 自定义多节点审批、逐节点留痕、类 BPM 引擎 | **部分响应或不响应** | 若产品侧为简化工作流，则**复杂 SOP** 建议对接企业现有审批/工单或通过集成实现 |
| 一键同步所有外部渠道 | **部分响应** | 通常需**按渠道对接**（官网/小程序/公众号等分别集成），分阶段交付 |
| 完全自定义元数据模型 | **部分响应** | 与产品字段与扩展能力对齐；深度 ECM 级建模可能需定制 |
| 数据看板（阅读量、互动、领域分布等） | **部分响应** | 依赖产品统计与**标签/分类体系**；部分指标需埋点或外接 BI |
| 版权到期自动下架、下架归档 | **部分响应** | 需规则与字段设计；「自动」依赖元数据维护机制 |
| 收藏、批量文件夹下载等 | **视版本** | 按路线图评估或分期 |

## 三、分期建议

### 阶段 M1（MVP，8～12 周，示意）

- 知识库 + DAM 上线，完成**历史内容迁移**（批量导入/脚本辅助）。  
- **角色、权限、审计日志**就绪。  
- **审核**：采用产品支持的**轻量流程**（如必要节点 + 医学部角色），满足「效率优先」。  
- **发布**：站点/帮助中心/内嵌页面先行；**API** 对接 1～2 个高优先级内部系统。  
- **搜索**：标题 + 标签；标签体系与首期导入方案确定。

### 阶段 M2

- **看板**：核心指标与领域分布（基于标签/分类治理）。  
- **版权与到期**：规则与字段落地，自动化下架/归档策略试运行。  
- **渠道扩展**：增加对接渠道或集成组件。

### 阶段 M3

- 评估 **高级工作流**、**更深元数据**、**更多渠道一键化** 等，与产品路线图及预算对齐。

---

## 四、暂不响应或需单独立项的部分（示例表述）

- **完全自定义 BPMN 级审批引擎**、与内部复杂法务/品牌串行链路 1:1 复刻：通常超出标准内容工具范畴，建议**集成**企业 BPM。  
- **无限制的 API 安全策略单元**（如细粒度到字段级的动态策略）：按企业安全规范**单独设计与评审**。  
- **直播与第三方独占媒资**：本期不纳入统一存储范围（与客户答疑一致）。

---

## 五、假设与依赖

- 客户侧指定**医学部审核责任人**与**可接受的内容分级标准**。  
- IT 提供**测试与生产环境**、**API 网关/鉴权**配合。  
- 迁移前完成**标签与分类共识**，否则看板与搜索效果受限。
