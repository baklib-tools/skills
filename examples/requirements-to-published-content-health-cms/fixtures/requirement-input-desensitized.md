# 客户需求说明（脱敏示例）

> **用途**：本文件随仓库分发，供在 Agent 对话中作为「客户材料」粘贴；**虚构企业与数据**，与任何真实客户无关。  
> **配套**：请在 Cursor 中同时打开团队本地的 **Baklib 产品知识库**（见 [上级 README](../README.md)），使模型能对照产品与本文需求。

**客户（虚构）**：华东某医疗健康科技公司「**康维医疗科技**」  
**场景**：建设内容中台，统一管理医学科普、患教、产品说明等数字内容，支撑多渠道分发与合规审核。

---

## 一、后端管理需求摘要

| 序号 | 需求项 | 说明 |
|------|--------|------|
| 1 | 文章管理 | 在线编辑，草稿，预览，权限 |
| 2 | 自定义内容审核工作流 | 不同文章类型对应不同审核流程 |
| 3 | 版权管理 | 版权到期自动下架 |
| 4 | 标签管理 | 标签增删改查，标题与标签搜索 |
| 5 | 内容发布管理 | 发布规则、定时发布等 |
| 6 | 分发渠道管理 | 分渠道、分平台管理 |
| 7 | 媒体文件管理 | 数字资产与媒体文件存储 |
| 8 | 自定义数据看板 | 内容总量、阅读、互动、领域分布等 |
| 9 | 标准 API 与文档 | 与集团内部系统对接 |
| 10 | 日志管理 | 操作日志 |
| 11 | 内容历史版本 | 可追溯、可复原 |
| 12 | 元数据管理 | 数字媒体资产元数据 |
| 13 | 内容归档 | 下架内容留存备查与分析 |

---

## 二、需求答疑（摘录）

1. 可接受**分阶段上线**，按工作量拆分。  
2. **直播**仍在第三方平台，媒资不迁入本期范围。  
3. 审核链路：**无需** IT/品牌/市场/法务多级；以业务流程图为准。  
4. 侧重**发布效率**，仅需**医学部**参与审核。  
5. 与 IT 集成事项可会后专项讨论。  
6. 标签可先期一次性导入，后续支持增删改查。  
7. 数据看板关注：总内容量、阅读、互动、**业务领域分布**（示例中为疾病领域）。  
8. 搜索首期：**标签 + 标题**即可。  
9. 账号与角色在后台按人配置，**无需对接 OA**。  
10. 内容量级约 **5000+** 条，容量 **40G+**。  
11. 现状：内容分散在部门网盘、旧后台、微信公众号等。

客户另提供过一份 PDF 版《内容中台产品需求说明》（**未在仓库内附链接**；演示中仅以本文与知识库为准）。

## 三、售前对照矩阵（示例，供方案引用）

以下为与产品能力对照的**示意标记**（o 满足 / - 部分满足 / x 不满足），具体结论须以**当前产品版本**与**本地知识库**为准。

- **角色权限**：搭建中台、分角色管理内容 — o / o  
- **内容生产**：富文本与协同、标签、导入 — 部分与产品组合相关（-）；统一入库、批量导入 — o；**高度自定义元数据模型** — 常为部分或需二次设计（x/-）  
- **内容审核**：**完全自定义 BPMN 级 SOP、逐节点留痕、审批通知** — 多为需集成或分期（x/-）  
- **发布与 API**：一键全渠道、API 矩阵 — 常为多产品组合与集成（-）  
- **内容管理**：标签体系、到期下架与归档、版本、定时发布、分享链接 — 多为 o/- 组合；收藏、批量文件夹下载、多维高级检索 — 视版本（x/-）  
- **数据看板**：完全自定义搭建 — 常为部分满足（-）  
- **系统设置**：审计日志、成员管理 — o；SSO — 常为 o/-；**审批流管理后台、API 安全策略细粒度** — 视产品与集成（x/-）

> 正式方案书中的条款须由 Agent 结合 **`baklib-workspace` 内文档**重新核实后输出，**不得**仅照抄本段。
