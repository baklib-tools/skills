# 本仓库协作约定（面向贡献者与自动化代理）

本文件说明 **baklib-tools/skills** 作为**公开开源仓库**在撰写技能、引用示例与协作时的全局规则。修改技能或新增 PR 前请先阅读。

## 仓库定位

- 本仓库的**首要目的**是向社区**发布与维护**与 **Baklib** 生态相关的 **Agent Skills**，供他人克隆、通过 **`npx skills add`**（skills CLI）等渠道安装，或复制到自己的项目中使用。
- **本仓库不是「在仓库里消费这些技能」的默认场景**：贡献者是在**补充技能目录与文档**，而不是假定某一产品会从本仓库根目录自动加载安装路径下的技能。使用者会在自己的项目里把技能放到**其工具链要求的技能目录**（路径因产品而异，以各工具文档为准）。
- 技能正文面向**公开读者**：应避免依赖未公开的内网路径、未发布的内部工具名或不可复现的私有流程。

## 技能存放位置（贡献时必读）

- **新增或编辑技能时**，一律放在仓库根目录的 **`skills/<skill-name>/SKILL.md`**。
- 每个技能为独立子目录，目录名与 frontmatter 中的 `name` 建议一致或明显对应。
- **不要**在本仓库根目录使用**使用者本地的技能安装路径**（由各工具文档规定）作为技能发布位置——发布应使用上文 **`skills/<skill-name>/`**；安装路径仅出现在使用者**自己的项目**中，且以所用工具文档为准。

## 技能配置读取约定

- 技能若需说明**如何获取配置**（密钥、Token、URL、功能开关等），应统一约定：从使用者工作区**根目录**下的 **`.config/`** 目录读取；不要在技能中假设其他默认配置目录（除非技能内对例外场景有单独说明）。
- 配置以**独立文件**存放，**文件名采用全大写**，与常见环境变量命名风格一致（仅大写字母、数字、下划线，如 `API_KEY`、`WEBHOOK_URL`），**文件名即该配置项的变量名**。
- 若需提供**可复制、可提交的示例**（格式说明、占位内容），使用 **`<变量名>.example`** 作为文件名（如 `API_KEY.example`、`WEBHOOK_URL.example`），与上述全大写变量名一一对应；文件内写入**与真实配置相同结构**的示例或虚构占位值，**不得**写入真实密钥或客户专属数据。代理或脚本读取生效配置时，**只应读取无 `.example` 后缀的文件**（除非技能明文约定例外）。
- 技能正文应说明：使用者将 `.example` 复制或对照后，在工作区 `.config/` 下创建**同名但去掉 `.example`** 的文件（例如由 `API_KEY.example` 得到 `.config/API_KEY`）并填入真实值；示例文件本身可放在 `skills/<skill-name>/` 随技能维护，或按技能说明放在使用者项目的 `.config/` 中且**仅提交 `*.example`**。
- 技能正文应给出可读的路径示例（如 `.config/API_KEY`），并继续遵守下文「脱敏与可移植性」：**只描述路径与命名规则，不得在仓库中提交真实秘密值**。
- 若技能同时支持从环境变量读取，须在 `SKILL.md` 中**写清优先级**（例如环境变量优先于 `.config` 下文件），避免代理或使用者对取值来源产生歧义。

## 脱敏与可移植性

1. **禁止**在技能中写入真实密钥、Token、Cookie、内网 URL、个人或客户的敏感路径。
2. **避免**绑定某一云厂商或某一脚本的**唯一**调用方式（例如特定 CLI 路径、私有环境变量名），除非该信息已在对外的官方文档中公开且确有必要；更推荐描述「将提示词交给你所使用的图像/导入工具或 API」这类**可替换**表述。
3. **内部工作区**（如私有 monorepo、个人 `read-and-writes`）里的技能若要迁入本仓库，需删除：仅内部存在的技能交叉引用、私有脚本路径、`团队/维护者` 等非必要身份信息；改为指向**本仓库内**已有技能或**公开文档链接**。
4. 示例中的产品名、域名、指标数字宜使用**占位符**或**虚构示例**（如 `example.com`、`XX%`），除非该示例已刻意作为品牌公开案例。

## 技能边界与责任范围

1. **每个技能应自成一体**：在 frontmatter `description` 中写清「做什么 / 何时触发」；正文中用简短章节声明**不做什么**，避免与脚本执行、计费账号、运维操作混淆。
2. **「提示词类」技能**（如图像模型提示词工程）**仅负责**自然语言提示词的结构、原则、避坑与复查清单；**不包含**运行任何脚本、调用任何 API、代用户配置密钥或选择供应商。实际出图、重试、费用由用户在其本地或自有流水线中完成。
3. **单次图像 API 请求通常无跨图记忆**：技能中若涉及「系列图、多版本」，应写明每条请求提示词须**自包含**全文，禁止假设模型能「引用上一张图」或「与上次保持一致」——除非明确写的是**同一次**多图/编辑接口的协议（与纯文生图单次调用区分）。
4. **「工具链类」技能**若存在，不应在另一技能中写死「必须先执行某私有脚本」；应允许用户仅阅读 `SKILL.md` 也能理解目标与产出物。

5. **工作流类技能须含确认门**：凡描述**多步编排**（如需求分析 → 方案 → 软文 → 渠道排版 → 配图 → 外部系统写入）的技能，须在 `SKILL.md` 中写明**执行前确认**——代理收到任务后**先**用简短编号列出**拟执行步骤**，并就**可选或高成本/高风险步骤**征求用户明确同意后再执行。至少应覆盖下列类型（与具体业务相关的可增列）：是否产出**推广/软文**、是否产出**公众号等渠道稿或 HTML**、是否**调用计费能力出图**（与仅撰写提示词或推导脚本区分）、是否**写入线上系统**（如 MCP/API）。**未确认前**，不得默认调用计费图像接口、不得擅自写入外部系统、不得假设用户需要完整流水线。**`description` 中宜点明「须先确认」**，便于发现与路由。

## 可选目录与配套文件（与 Agent Skills 规范对齐）

开放规范（见 [agentskills.io/specification](https://agentskills.io/specification)）将技能定义为一个目录，**至少**包含 `SKILL.md`；其余为可选，用于**渐进式披露**（启动时多用 frontmatter 做发现，激活技能后再读正文与附件）。

| 路径 | 用途 |
|------|------|
| **`references/`** | 可选。放**深度参考**（可多文件、按主题拆分）。代理**按需读取**，避免主文件一次塞满长表与全量语法。 |
| **`reference.md`** | 可选。单文件深度参考时的命名；与本仓库已有技能（如 `baklib-intake-assistant`）保持一致即可。 |
| **`scripts/`** | 可选。可执行脚本；技能正文应说明如何调用与安全边界。 |
| **`assets/`** | 可选。模板、示意图等静态资源。 |

**与「技能根目录平铺的 `.md`」的关系**：已有技能若将 `routing-guide.md`、`local-mirror.md` 等放在技能根目录，**可保留**；**新增技能**若有**多份**补充文档，**优先**收入 **`references/`**，便于与规范用语及跨工具文档一致。

**主 `SKILL.md` 建议**：保留 frontmatter、`description` 所覆盖范围内的**执行步骤 / 路由 / 速查**；**大段语法表、长清单、可独立成章的说明**优先迁入 `references/` 或单文件 `reference.md`。规范与业界实践均建议控制主文件篇幅（例如主说明宜明显短于「全量参考」之和），避免激活单技能时占用过多上下文。

## 文风与维护

- 使用简体中文或中英混排时保持术语一致；第三方产品名以官方为准。
- 大段重复内容优先拆到同目录下的 **`references/`**、**`reference.md`** 等文件（由主 `SKILL.md` 以链接引用），避免单文件过长。
- 更新技能时同步更新根目录 [README.md](README.md) 中的技能表（若有新增或重命名）。

## 与本仓库其他文件的优先级

- 具体技能的细则以各 `SKILL.md` 为准；**本文件**约束存放位置、配置读取约定、公开性与责任边界。若某技能需例外说明，应在技能内显式写出适用范围。
