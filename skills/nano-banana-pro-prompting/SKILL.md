# Nano Banana Pro 提示词工程

> 面向 **Gemini 3 Pro Image**（社区常称 **Nano Banana Pro**）的提示词写法  
> 核心理念：**分步叙述、超具体描述、严格复查**

## 技能范围（必读）

- **本技能只解决一件事**：如何**撰写与优化**交给图像模型的**提示词正文**（自然语言）。
- **不包含**：运行命令、调用 HTTP API、配置密钥、选择云厂商或本地脚本路径；也不对任何仓库内外的「生成图像脚本」负责。
- 实际出图时，请将最终提示词复制到你使用的**任意**图像生成产品、插件或自建工具中执行。

## 模型信息（公开产品名）

| 属性 | 说明 |
|------|------|
| 社区俗称 | Nano Banana / Nano Banana Pro |
| 正式名称 | Gemini 3 Pro Image（以 Google 官方命名为准） |
| 常见能力 | 复杂指令遵循、图内文字渲染、**推理（Thinking）**、风格一致性（具体以所用平台说明为准） |

官方文档称 [Gemini 3 Pro Image](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/models/gemini/3-pro-image) 面向高难度图像生成并融入**先进推理能力**；模型能力表中标明支持 **Thinking**。因此在**同一条分步提示词末尾**增加「生成前内化自检」步骤，与「模型会先推理再出图」的设计**方向一致**，属于合理提示工程；是否显著减少错字/扭曲**无公开量化保证**，需与下文「效果边界」一并理解。

分辨率、思考过程是否对用户可见、计费等**因平台而异**；请在实际使用的产品文档中确认。

## 核心提示词原则

### 原则 1：分步叙述优于关键词堆砌

模型更擅长遵循**有顺序的自然语言指令**，而不是松散关键词列表。

**不推荐**：`clean, modern, tech, blue, infographic...` 一串无结构词。

**推荐**：用 `Step 1:`、`Step 2:` 逐步写清背景、主体、次要元素、文字与风格。

### 原则 2：超具体描述（Hyper-Specific）

细节越具体，可控性越高。把「现代 UI」写成「卡片式布局、圆角、主按钮为青色白字」等。

### 原则 3：说明用途与情绪

说明这张图用于**痛点对比**还是**功能说明**，会显著影响构图与配色。

### 原则 4：语义化「负面」表述

少用 `no text` 这类可能被误读的否定堆叠；改用「画面留白充足、仅保留 Step 7 列出的可见文字」等正向描述。

### 原则 5：用摄影/镜头语言控制构图

如：`wide-angle`、`isometric 45°`、`soft three-point lighting`、`shallow depth of field`。

### 原则 6：单次请求自包含（无跨图上下文）

典型 **文生图 API 每次调用只处理当前这一条提示词**，不会带上你「上一张图」的生成历史（除非你使用的是**明确支持多轮对话/参考图**的产品形态，且协议里写清了如何传入前序内容——本技能默认 **单次纯文本提示词直调 API**）。

- **无效且有害**：写「与上一张一致」「同前图角色」「MUST be identical to the previous shot」「see Shot 1」——模型**看不到**上一张，这类句子**不提供可执行信息**，等于没写。  
- **正确做法**：需要多张图风格/角色/图标一致时，在**你自己的侧**维护一份「主档案」（角色段落、图标列表、风格句），**每发一次 API 都把完整段落粘贴进当次提示词的 Step 4/5/8**；允许重复、不允许省略后靠「引用前文」。  
- **同一条提示词内**若包含多面板/分屏（一张图里左右两栏），可在**该条内部**写「左右使用同一角色描述如下：……」，这是**同一请求内的自洽**，与跨请求无关。

## 标准结构：分步模板（Step 1–8 + Step 9 模型侧自检）

每张图的提示词可按以下块组织（可按场景删减，但建议保留 **Step 7 / 8 / 9** 的职责划分）：

```
Step 1: [背景与氛围] — 背景、渐变、光、情绪
Step 2: [主视觉] — 中心图、界面或产品主体
Step 3: [辅助元素] — 连线、装饰、数据暗示
Step 4: [图标] — 每个图标的造型、颜色、标签文案
Step 5: [角色] — 外貌、服装、表情、姿势（每张 API 请求都须写全；勿写「同前图」「与上一张一致」）
Step 6: [状态符号] — 对勾、问号、光效等
Step 7: [图内文字] — 标题、副标题、数据（唯一信源，列清内容与大致位置）
Step 8: [技术与风格] — 画幅比例、渲染风格、禁止项（如不要出现 HEX 码作为图内字）
Step 9: [输出前自检 — 利用推理] — 在生成最终像素前，在内部完成核对（见下节英文示例与说明）
```

### Step 9 写什么（与「Thinking」对齐）

你的意图是：**在提示词里显式要求模型在出图前做一次「错误/异常」核对**，利用 Nano Banana / Gemini 3 Pro Image 的推理能力，而不是只靠 Step 1–8 堆描述。

[《Gemini 3 prompting guide》](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/start/gemini-3-prompting-guide) 建议：复杂任务可采用**分步验证**（先验证、再生成）；且**最关键的限制宜放在指令后部**。把 **Step 9 放在 Step 8 之后**，相当于在「技术约束」之后再叠一层**内化检查清单**，与官方「约束置后、分步验证」的思路一致。

**可复制英文骨架（按 Step 7 实际文案改写括号内要点）：**

```
Step 9: Before finalizing the image, internally verify (do not print this checklist as visible text):
- Every Chinese and English string matches Step 7 exactly — same characters, same spelling, simplified Chinese where required, no duplicate or conflicting labels.
- No color hex codes, API tokens, or meta-instructions appear as readable text unless explicitly listed in Step 7.
- Text is legible, not warped along perspective; labels sit on clear flat surfaces where possible.
- If any check fails, adjust the composition to satisfy Step 7 and Step 8 before completing the render.
```

说明：

- **「Internally verify」**：与「先思考再出图」一致；模型是否在界面里展示思考过程**取决于平台**，自检**不保证**以文字形式回显。
- Step 9 **不能替代**成品图的人眼（或视觉模型）复查：推理仍可能漏检，见下「效果边界」。

### 效果边界（必读）

| 预期 | 说明 |
|------|------|
| **可能有效** | 把核对清单放在**最后一步**，引导推理阶段优先对齐 Step 7/8，减少明显错字、指令泄露、重复标签。 |
| **不保证** | 一次出图即像素级完美；复杂透视上的字仍可能歪、糊。 |
| **与生成后复查的关系** | Step 9 = **提示词内的模型侧自检**；「生成后复查」= **对输出文件再验**。二者互补，不是二选一。 |

涉及**具体品牌或角色档案**时，应从你的工作区或品牌规范中**自行**维护文本，并在**每一次**请求中**整段粘贴**进提示词；本技能不提供私有素材路径。

## 避坑指南

### 1. 颜色值被画进图里

HEX（如 `#0D9488`）可能被当成图内文字。优先用自然语言颜色名；若必须标注 HEX，放在技术段并声明「最终画面不得显示色值字符串」。

### 2. 同一概念两种写法

在 Step 7 **集中**列出所有固定文案，并声明全文一致；避免 `/docs` 与 `/doc` 混用。

### 3. 重复指标或标题

为每条文案指定**唯一位置**，并在 Step 8 声明不重复出现。

### 4. 中文变体错误

若需要大陆简体，在提示词中明确：`Use Mainland China Simplified Chinese for all Chinese text in the image`，并避免繁体、港台用词（按你的发布地区调整表述）。

### 5. 英文说明被当成图内字

只有 Step 7 中**明确要显示**的短语用引号；技术形容词不要加引号，避免被渲染成可见文字。

---

## 多张图之间的「一致性」怎么做（无跨请求记忆）

前提：**每一次调用 API 都是独立的**，模型不会记住你上一张图里写了什么。

| 错误写法（勿用） | 原因 |
|------------------|------|
| 「角色与上一张相同」「沿用 Shot 1 设定」「MUST be identical to previous」 | 当前请求里**没有**上一张的信息，指令为空壳。 |
| 「保持系列风格一致」且不展开 | 过于抽象，没有可执行细节。 |

| 正确做法 | 说明 |
|----------|------|
| **主档案 + 逐次全文粘贴** | 在本地文档维护「角色全文」「图标清单」「Step 8 风格段」；生成**每一张**图时，把这三段**完整复制**进当次提示词，再只改本张特有的构图/文案/表情。 |
| **同一次请求内的多区域** | 若一张图里画左右对比，可在**本条**里写两遍同一角色描述，或写「LEFT/RIGHT use the same character: [完整描述]」——信息仍在**单次提示词内**。 |
| **仅状态变化** | 多张图只有「散乱 vs 聚合」等差异时，仍须在每张提示词里**重复**相同图标与角色的完整描述，只改状态词。 |
| **`MUST be identical` 何时可用** | 仅当**同一条**提示词里要求图内两个元素一致（例如左右两个徽章同一风格），或你已在**本提示词中**重复写出了两段相同描述并声明一致——**不要**用于指涉「上一张 API 的图」。 |

人工侧可对齐：生成后把新图与**主档案**或**上一张导出文件**对比，不满意则改**下一次请求的提示词**（加厚描述），而不是指望模型「记得」系列约束。

## 生成前复查清单（简版）

**内容**：简体要求、URL/路径拼写统一、无重复文案、数字与指标只出现一次。  
**提示词卫生**：HEX 不当作默认图内字；非展示用语不加引号。  
**风格**：画幅、渲染关键词是否与**本提示词内** Step 8 一致。  
**多图项目（人工核对）**：若你在做系列，发下一 API 前是否已把**主档案全文**贴进本次提示词；**不要**假设模型记得上一张。  
**多图叙事**：若成对「痛点/解决」，两张是否分别**各自写全**角色与风格段，且情绪对比写在各自提示词里。

出图后还必须做下文 **「生成后复查」** 一节。提示词内 **Step 9** 可降低部分错误概率，**不能**替代对成品的检查。

## 图内文字要点

- 用引号标出**必须原样显示**的短语。  
- 用描述表达字体气质（`clean sans-serif`），避免依赖特定版权字体名。  
- 控制单张图文字组数（例如不宜过多），以免糊字或拥挤。

### 减轻「错字、扭曲、变形」的写法（预防）

图内字常见问题包括：笔画断裂、透视把字「拉歪」、小区域塞字导致糊成一团、中英文混排挤压。可在 Step 7/8 中酌情加入：

- **少字优先**：同一画面文字组数越少，单字越易清晰；能删则删，能短则短。  
- **平面可读**：要求标题/标签贴在**近似平面**或清晰界面上（如屏幕、标牌），避免强透视大字。  
- **留白与对比**：声明文字区域有足够留白、高对比（如白底深字），避免与复杂纹理叠在一起。  
- **禁止把说明性英文当图内字**（与上文避坑一致）：减少模型把指令词画进画面。

## 生成后复查（必做）：文字、内容与评分

生成前清单**不能替代**出图后的检查；模型仍有概率产出错字、歪字或与 Step 7 不一致的文案。**复查由人工查看成品图完成**；若你的工作区有多模态/视觉模型，也可让其按下列项逐项描述画面，再对照判断。

### 1. 文字专项（优先查）

| 检查项 | 不合格示例 | 处理方向（改提示词时） |
|--------|------------|------------------------|
| 可读性 | 模糊、过细、被遮挡、与背景糊在一起 | Step 7 减字或改位置；Step 8 强调对比度、平面展示 |
| 字形扭曲 | 透视变形、沿曲面拉伸、波浪状字 | Step 7 要求「flat label / front-facing text」；减弱透视描述 |
| 内容与 Step 7 不一致 | 错字、漏字、多字、繁简混用 | 直接修订 Step 7 文案与「exactly as quoted」类约束 |
| 指令泄露 | 画面出现英文技术说明、提示词碎片 | Step 7/8 重申「only quoted text visible」；检查误加引号 |
| 重复/矛盾 | 同一指标两处不同数字 | 回到 Step 7 唯一位置声明 + Step 8「each label once」 |

### 2. 内容与需求符合度

- 画面是否表达**原定叙事**（痛点/方案/架构等）？主体、图标数量与关系是否对？  
- 若与需求不符：区分是**文案问题**（偏 Step 7）、**构图/元素**（偏 Step 1–3）、**角色**（偏 Step 5）、**整体风格**（偏 Step 8）。

### 3. 简易内部评审（打分）

用一张表快速决定是否**需要重跑**（不必追求数学精确，目的是减少主观「差不多」）：

| 维度 | 说明 | 分档建议 |
|------|------|----------|
| **文字** | 可读、无错、无扭曲、与 Step 7 一致 | 不满意则**必须**修订后重试 |
| **内容** | 叙事与信息符合需求与品牌约束 | 不满意 → 改对应 Step 文案或结构 |
| **画面** | 构图、风格可接受；若属系列项目，与**主档案**或既定参考是否对齐（人工比对） | 不满意 → 回到 Step 1–2 或 Step 8，并在**本条提示词内**补全描述 |

**通过阈值（建议）**：文字维度须达标；内容与画面至少一项可接受，或两项均为小瑕疵且可接受。若**文字项不达标**，不要强行交付，应进入下一轮修订。

### 4. 未通过时：回到哪一步（修订路由）

按**主要问题类型**定向改提示词，再生成；避免无目的地全文重写。

| 主要问题 | 优先回到 |
|----------|----------|
| 图内字错、漏、多、歪、糊 | **Step 7**（必要时 **Step 8**；已写 **Step 9** 仍不佳时，收紧 Step 7 文案或加强 Step 9 核对项） |
| 整体风格、比例、禁止项未满足 | **Step 8** |
| 背景、主视觉布局、信息层级不对 | **Step 1–3** |
| 角色外貌/衣着与**主档案**或参考图不一致 | **Step 5**：在**下一次请求的提示词里**写全角色段，勿写「同上一张」 |
| 图标与**主档案**不一致 | **Step 4**（及 Step 7）：全文重复图标清单 |

可多次循环：**生成 → 生成后复查 → 定向改 Step → 再生成**，直到评分可接受或达到团队约定的最大重试次数。

---

## 成本与迭代（通用建议）

- **顺序**：生成前清单 → 提示词含 **Step 9** →（试跑一张）→ 出图 → **生成后复查与评分** → 不通过则按上表改 Step 再生成。  
- 宁可多轮**定向小改**，也不要在同一提示词上盲目重复调用。  
- 系列多图时：先用**文字最少**的一张定风格，把定稿的 **Step 4/5/8 段落**存成主档案；**每发一张新图都整段粘贴**，再改该张独有内容。
